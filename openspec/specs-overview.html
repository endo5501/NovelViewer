エグゼクティブサマリー

NovelViewer は、Web小説サイト（小説家になろう・カクヨム）および青空文庫から作品を取得し、ローカルで快適に閲覧するための デスクトップアプリケーションです。Flutter + Riverpod を基盤とし、macOS / Windows / Linux を対象に、縦書き・横書き両対応の読書体験、 ローカルTTS（音声合成）による読み上げ、LLM による登場人物・キーワードのネタバレ制御付き要約という 3 つの大きな柱を持ちます。

本ドキュメントは openspec/specs/ 配下の 80 件の capability 仕様（計 487 要求）を、 実装単位のままではなく 9 つの機能ドメインに再編成し、各機能の目的と規模を一望できるようにしたものです。 正式な規範（SHALL／MUST など）は各 spec.md を参照してください。本書は理解のための要約・再構成です。

主要コンセプト・用語集

仕様全体を通じて繰り返し登場する横断的な概念・技術用語です。

Riverpod Provider
状態管理の中核。UIは原則 Provider 経由でのみ状態を参照し、各ビューアは対応 capability の Provider のみに依存する（疎結合の徹底）。
Dart FFI / Isolate
ネイティブ C++ TTS エンジンを呼ぶための外部関数インタフェース。重い合成処理はバックグラウンド Isolate で実行し UI をブロックしない。
qwen3-tts.cpp / Piper
2 種のローカル TTS エンジン。Qwen3-TTS は音声クローン対応の高品質系、Piper は軽量系。ユーザが切替可能。
スナップショット / covered_up_to_episode
LLM 要約のネタバレ制御単位。「どの話数まで読んだ状態の要約か」を上限として記録し、ネタバレあり/なしを切り替える。
ルビ（<ruby>）
HTML ルビタグ。横書きでは基底文字の上、縦書きでは右側に振り仮名を描画。検索・TTS・要約マークは基底文字に対して作用。
禁則処理（kinsoku）
縦書きレイアウトの行頭・行末禁則。押し出し方式で改行位置を調整し、フォント実測値からページ分割を行う。
セグメント（segment）
TTS 音声の最小単位。文ごとに合成・保存され、フォルダ別 SQLite（tts_audio.db）に永続化される。
SharedPreferences / secure_storage
設定の永続化先。非機密はSharedPreferences、APIキー等の機密は flutter_secure_storage に保存。
スナップショット三つ組キー
(folder_name, word, covered_up_to_episode)。LLM要約キャッシュ word_summaries の一意キー。
隣接ファイル導出 / intent
「次話/前話」遷移の基盤。開始位置ヒント（fromStart/fromEnd）を渡し、横書き・縦書き共通で扱う。

システム構成

UI は固定 3 カラム（左：ファイル/ブックマーク/履歴タブ・中央：テキストビューア・右：検索＆要約パネル）を基本とし、 その背後で TTS・LLM・ダウンロード・永続化の各サブシステムが Riverpod Provider を介して連携します。 下図は機能ドメイン間の依存関係を俯瞰した推定ビューです（個々の spec を統合して作図）。

主要ワークフロー

① 小説のダウンロードと閲覧

URL からの取得 → エピポードキャッシュによる増分判定 → ローカル保存 → ファイルブラウザ表示 → 読書再開という基本動線です。

② TTS ストリーミング再生

文分割（ルビ除去・辞書変換）→ Isolate での逐次合成 → セグメント永続化 → 再生・ハイライト・自動ページ送り。

③ LLM 要約（ネタバレ制御）

語句選択 → 解析範囲（話数上限）決定 → 事実キャッシュ再利用 → チャンク抽出・蒸留 → 最終要約 → スナップショット保存・本文マーク。

ドメイン別 機能一覧

注意点・未確定事項

Purpose が TBD の仕様（5件）

以下の仕様は ## Purpose が「TBD - created by archiving change…」のままアーカイブされており、本書の概要は 各 spec の Requirement 名から要約しています（推定要約で明示）： windows-installer ／ app-update-check ／ database-recovery ／ llm-summary-context-menu-trigger ／ llm-summary-hover-popup ／ voice-file-rename ／ voice-file-drag-drop。 正確な目的記述は今後の更新で補完すると良いでしょう。

TTS 重心と保守上の含意

TTS 系が全体の約 40%（197 要求）を占め、ネイティブ C++・FFI・GPU バックエンド（Vulkan/Metal）・CI でのDLLビルドまで広がります。 変更時はエンジン基盤（ドメイン4）→ モデル/設定（5）→ 再生/生成（6）の依存方向を意識すると影響範囲を見積もりやすくなります。 特に tts-native-engine・tts-streaming-pipeline・tts-audio-storage は多数の機能が依存する土台です。
