## Purpose

Distinguish novel folders from organizational folders within the library, and provide create, rename, move, and delete operations for organizing novels into a folder hierarchy while keeping per-folder database handles consistent.
## Requirements
### Requirement: 小説フォルダと整理フォルダの判別

システムは、あるフォルダの basename が `novels` テーブルの `folder_name` 集合に含まれる場合にそのフォルダを「小説フォルダ」とし、含まれない通常ディレクトリを「整理フォルダ（organizational folder）」として扱うものとする (SHALL)。この判別はライブラリルートからの深さに依存してはならない (MUST NOT)。

#### Scenario: basename が folder_name に一致するフォルダは小説フォルダ
- **WHEN** あるフォルダの basename が `novels` テーブルの `folder_name` のいずれかに一致する
- **THEN** そのフォルダは小説フォルダとして扱われ、データベース上のタイトルで表示され、小説向けの右クリックメニューが有効になる

#### Scenario: folder_name に一致しないフォルダは整理フォルダ
- **WHEN** あるフォルダの basename が `novels` テーブルのどの `folder_name` にも一致しない
- **THEN** そのフォルダは整理フォルダとして扱われ、フォルダ名がそのまま表示される

#### Scenario: 入れ子の小説フォルダも深さに関わらず判別される
- **WHEN** 整理フォルダの配下（ライブラリルートから2段以上深い位置）に `folder_name` と一致する小説フォルダが存在する
- **THEN** その小説フォルダはルート直下にある場合と同様に小説フォルダとして判別される

### Requirement: 整理フォルダの作成

システムは、ファイルブラウザで現在表示中のディレクトリ内に新規の整理フォルダを作成する操作を提供するものとする (SHALL)。フォルダ名は OS のファイル名制約に反する無効文字を含んではならず (MUST NOT)、同じ親ディレクトリ内に同名のフォルダ・ファイルが既に存在する場合は作成を拒否しなければならない (MUST)。

#### Scenario: 有効な名前で整理フォルダを作成
- **WHEN** ユーザーが現在表示中のディレクトリで有効な名前を指定して新規フォルダ作成を実行する
- **THEN** 指定した名前の整理フォルダが現在のディレクトリ内に作成され、ファイル一覧に表示される

#### Scenario: 既存と同名のフォルダ作成は拒否される
- **WHEN** ユーザーが、現在のディレクトリに既に存在するフォルダ名と同じ名前で新規フォルダ作成を実行する
- **THEN** フォルダは作成されず、名前が重複している旨のエラーが表示される

#### Scenario: 無効文字を含む名前は拒否される
- **WHEN** ユーザーが OS のファイル名として無効な文字を含む名前で新規フォルダ作成を実行する
- **THEN** フォルダは作成されず、名前が無効である旨のエラーが表示される

### Requirement: 整理フォルダのリネーム

システムは、整理フォルダに対するリネーム操作で実ディレクトリ名を変更するものとする (SHALL)。同じ親ディレクトリ内に同名のフォルダが既に存在する場合はリネームを拒否しなければならない (MUST)。小説フォルダに対するリネームは実ディレクトリ名を変更せず、表示タイトルのみを更新する従来の挙動を維持するものとする (SHALL)。

#### Scenario: 整理フォルダのリネームで実ディレクトリ名が変わる
- **WHEN** ユーザーが整理フォルダをリネームし、新しい有効な名前を指定する
- **THEN** 実ディレクトリ名が新しい名前に変更され、ファイル一覧に新しい名前で表示される

#### Scenario: 小説フォルダのリネームは表示タイトルのみ変更する
- **WHEN** ユーザーが小説フォルダをリネームし、新しいタイトルを指定する
- **THEN** 実ディレクトリ名（`folder_name`）は変更されず、データベース上の表示タイトルのみが更新される

#### Scenario: 整理フォルダのリネームで同名衝突は拒否される
- **WHEN** ユーザーが整理フォルダを、同じ親ディレクトリ内に既に存在するフォルダ名へリネームしようとする
- **THEN** リネームは行われず、名前が重複している旨のエラーが表示される

### Requirement: フォルダの移動

システムは、小説フォルダおよび整理フォルダを、ライブラリ内の別のフォルダへ移動する操作を右クリックメニューから提供するものとする (SHALL)。移動はフォルダの葉の名前（小説フォルダの場合は `folder_name`）を変更せず、親パスのみを変更するものとする (MUST)。移動先には移動元自身およびその子孫を選択できてはならず (MUST NOT)、移動先に同名のフォルダが既に存在する場合は移動を拒否しなければならない (MUST)。

#### Scenario: 小説フォルダを整理フォルダへ移動する
- **WHEN** ユーザーが小説フォルダを右クリックして「移動」を選び、移動先の整理フォルダを指定する
- **THEN** 小説フォルダが移動先の整理フォルダ内へ移動し、葉のフォルダ名（`folder_name`）は変わらず、データベースの紐付け・読書進捗・TTS音声データは維持される

#### Scenario: 整理フォルダを別の整理フォルダへ移動する
- **WHEN** ユーザーが整理フォルダを右クリックして「移動」を選び、移動先の整理フォルダを指定する
- **THEN** 整理フォルダがその中身ごと移動先へ移動する

#### Scenario: 移動先に同名フォルダが存在する場合は拒否される
- **WHEN** ユーザーが、移動先に既に同名のフォルダが存在する状態で移動を実行する
- **THEN** 移動は行われず、移動先に同名フォルダが存在する旨のエラーが表示され、既存フォルダは上書きされない

#### Scenario: 自分自身および子孫への移動は選択できない
- **WHEN** ユーザーが整理フォルダの移動先を選ぶ
- **THEN** 移動元のフォルダ自身およびその子孫フォルダは移動先候補として選択できない

### Requirement: 移動先選択ダイアログ

システムは、移動操作時にライブラリルートを根とする整理フォルダ階層を表示するツリー選択ダイアログを提供するものとする (SHALL)。ダイアログはライブラリ内のフォルダのみを表示し、ライブラリ外のディレクトリを選択できてはならない (MUST NOT)。ダイアログには整理フォルダおよびライブラリルートを移動先候補として表示し、小説フォルダの内部を移動先として表示してはならない (MUST NOT)。

#### Scenario: ダイアログがライブラリ内のフォルダ階層を表示する
- **WHEN** ユーザーが移動操作を開始する
- **THEN** ライブラリルートを根とする整理フォルダの階層がツリー表示され、移動先として選択できる

#### Scenario: ライブラリ外は選択できない
- **WHEN** 移動先選択ダイアログが表示されている
- **THEN** ライブラリルートより上位のディレクトリは表示されず、選択できない

### Requirement: 整理フォルダの削除

システムは、空の整理フォルダのみ削除を許可するものとする (SHALL)。中に小説フォルダ・ファイル・子フォルダが残っている整理フォルダの削除は拒否しなければならない (MUST)。

#### Scenario: 空の整理フォルダを削除する
- **WHEN** ユーザーが空の整理フォルダに対して削除を実行する
- **THEN** その整理フォルダが削除され、ファイル一覧から消える

#### Scenario: 空でない整理フォルダの削除は拒否される
- **WHEN** ユーザーが、中にフォルダまたはファイルが残っている整理フォルダに対して削除を実行する
- **THEN** 削除は行われず、フォルダが空でないため削除できない旨のメッセージが表示される

### Requirement: 移動・削除時のデータベースハンドル整合

システムは、フォルダの移動・リネーム・空フォルダ削除によって絶対パスが変化または消滅する際、対象フォルダに紐づく per-folder データベースハンドル（TTS音声DB・TTS辞書DB・エピソードキャッシュDB・**小説データDB(`novel_data.db`)**）を解放し、操作後の状態と整合させるものとする (SHALL)。ハンドルの解放は、実ファイルシステム操作（`Directory.rename` / `delete`）を開始する**前に**、各ハンドルの `close()` の完了を待って（await して）行わなければならない (MUST)。

per-folder データベースハンドルは、open/close を所有する**ハンドルレジストリ**が管理するものとする (SHALL)。移動・リネーム・空フォルダ削除・小説削除の各フローは、レジストリの awaitable な `closeAll(folder)` を**唯一の解放API**として経由してハンドルを解放するものとする (SHALL)。`closeAll(folder)` は、対象フォルダの4ハンドルの `close()` 完了を待ってからレジストリのキャッシュより除去するものとする (SHALL)。解放キーは `folderDbKey` で正規化し、パス区切りの綴り（`/` と `\`）が異なっても同一フォルダが同一ハンドルへ解決されなければならない (MUST)。ダウンロード専用の `closeEpisodeCache` はエピソードキャッシュDBのみを閉じ、`novel_data.db` を含む他のハンドルを閉じてはならない (MUST NOT)。

Riverpod の per-folder DB provider はレジストリ上の薄いビューとし (SHALL)、widget 層その他の消費者がハンドルの `close()` 順序を直接振り付けてはならない (MUST NOT)。`ref.invalidate` のみによる解放（`onDispose` の `close()` が await されない fire-and-forget）に依存してはならない (MUST NOT)。

ハンドル解放（`closeAll`）の後、システムは対象フォルダの thin-view provider を無効化し、provider のキャッシュをレジストリの最新状態（evict 済み）に追従させなければならない (MUST)。非 autoDispose の `Provider.family` は解決済みラッパーをキャッシュするため、無効化を怠ると provider が evict 済み（close 済み）ハンドルを返し続け、接続ゲートの再 open でレジストリ管理外の接続が開いてファイルを再ロックし得る。解放（`closeAll`）と provider 無効化は単一のヘルパーに束ね、各解放フローがこれを経由するものとする (SHALL)。

これにより、Windows で SQLite ファイルが排他ロックされたままファイル操作とレースし、操作が無言で失敗することを防ぎ、新しい per-folder DB 消費者を追加しても locked-DB バグが再発しないことを保証する。現在開いているフォルダがその配下にある場合、システムは現在のディレクトリを操作後の状態に追従させるか、操作を安全に拒否するものとする (SHALL)。

#### Scenario: 開いていない小説フォルダの移動でハンドルが解放される
- **WHEN** 現在開いていない小説フォルダを移動する
- **THEN** その小説フォルダに紐づく per-folder データベースハンドルが `closeAll(folder)` 経由で解放され、移動後に新しいパスで再取得される

#### Scenario: ファイル操作の前に全ハンドルの close が完了している
- **WHEN** 小説フォルダの移動・リネーム・空フォルダ削除のいずれかが実行される
- **THEN** TTS音声DB・TTS辞書DB・エピソードキャッシュDB・小説データDB(`novel_data.db`) の4ハンドルの `close()` が完了してから、`Directory.rename` / `delete` が呼び出される
- **AND** `close()` の完了を待たずにファイル操作が開始されることはない

#### Scenario: ハンドル解放はレジストリの closeAll を唯一の経路とする
- **WHEN** 移動・リネーム・空フォルダ削除・小説削除のいずれかのフローが per-folder ハンドルを解放する
- **THEN** 解放はレジストリの `closeAll(folder)` を経由して行われる
- **AND** widget 層が個別ハンドルの `close()` 順序を直接振り付けることはなく、`ref.invalidate` のみによる fire-and-forget 解放も行われない

#### Scenario: ダウンロードフローは小説データDBを閉じない
- **WHEN** ダウンロードフローが `closeEpisodeCache(folder)` を呼ぶ
- **THEN** エピソードキャッシュDBのみが閉じられ、`novel_data.db` を含む他のハンドルは閉じられない

### Requirement: folderDbKey 正規化のクロスプラットフォーム検証

「移動・削除時のデータベースハンドル整合」要件が定める `folderDbKey` の正規化不変条件（冗長な等価パスが同一フォルダ・同一ハンドルへ解決されること）を検証するテストは、特定OS固有のパス形式をハードコードしてはならない (MUST NOT)。検証は、`package:path` のホストOSセパレータ（`p.separator` / `p.join` 等）を用いて等価パスのペアを構築し、Windows・macOS（POSIX）の双方で同一の意図を成立させなければならない (SHALL)。これにより、`folderDbKey` がプラットフォーム文脈で正規化する仕様を、いずれのOSでも回帰として検知できる。

#### Scenario: 等価な冗長パスはホストOSで同一ハンドルへ解決される

- **WHEN** レジストリに対し、ホストOSのセパレータで構築した基準パスと、`..` を含む同義の冗長パス（例: `<base>/n1/../n1` 相当）で同一種別のハンドルを取得する
- **THEN** `folderDbKey` による正規化で両パスが同一キーへ解決され、同一のハンドルインスタンスが共有される
- **AND** この検証は Windows・macOS の双方でパスする

#### Scenario: closeAll が冗長パスでも正規化キー経由でハンドルへ到達する

- **WHEN** 基準パスで開いたハンドルに対し、`..` を含む同義の冗長パスで `closeAll` を呼ぶ
- **THEN** 正規化キーが一致し、対象ハンドルの `close()` が実行される
- **AND** この検証は Windows・macOS の双方でパスする

#### Scenario: テストはOS固有のパス形式をハードコードしない

- **WHEN** `per_folder_db_registry_test.dart` の正規化検証テストを走査する
- **THEN** `C:\...` のような Windows 固有絶対パスや特定OSのセパレータをハードコードした入力が、クロスプラットフォーム前提のテストに含まれない
