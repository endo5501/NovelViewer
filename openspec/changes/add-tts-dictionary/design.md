## Context

NovelViewerのTTS機能は各エピソードのセグメントをSQLiteで管理する。ユーザーは編集ダイアログからセグメントテキストを手修正できるが、人名・固有名詞などの特殊な読みは毎エピソード・毎セグメントで手修正が必要となっている。辞書機能はこの繰り返し作業を解消する。

現在のフロー:
- `TtsStreamingController.start(text, ...)` → `TextSegmenter.splitIntoSentences(text)` でセグメント分割 → TTSエンジンへ渡す
- セグメントテキストの手修正は `tts_audio.db` の `tts_segments.text` に保存される

## Goals / Non-Goals

**Goals:**
- 小説フォルダごとに表記→読みのペア辞書を永続化する
- TTS入力テキスト生成時（`TtsStreamingController.start()` と `TtsEditController` のオンデマンド生成）に辞書変換を自動適用する
- TTS編集ダイアログから辞書を管理できるUIを提供する（追加・編集・削除）

**Non-Goals:**
- グローバル辞書（全小説共通）の管理
- 正規表現による変換パターン
- 変換の優先順位制御（単純な最長一致マッチで十分）
- 形態素解析との連携

## Decisions

### 決定1: 辞書用の独立DBファイル（`tts_dictionary.db`）

**採用**: 既存の `tts_audio.db` に `tts_dictionary` テーブルを追加するのではなく、`tts_dictionary.db` を別ファイルとして作成する。

**理由**:
- `tts_audio.db` はエピソード単位の音声データを管理し、ロールバックや削除の単位が異なる
- 辞書は小説全体に対して有効であり、エピソードのDBとライフサイクルが異なる
- 将来的な辞書のインポート/エクスポートが独立ファイルで容易になる

**代替案**: `tts_audio.db` に辞書テーブルを追加 → テーブルの責務が混在し、DBスキーマ変更が既存データに影響するリスクがある

### 決定2: テキスト変換の適用タイミング

**採用**: `TtsStreamingController.start()` の入力 `text` に対して辞書変換を適用する。また `TtsEditController` でオンデマンド生成する際のセグメントテキストにも適用する。セグメント編集画面に表示するテキスト（`tts_segments.text`）は変換前のオリジナルのまま保存する。

**理由**:
- DBに保存するテキストは人間が読む表記のままにしておくことで、ユーザーが辞書を更新しても既存セグメントの表示が壊れない
- 変換はTTSエンジンへ渡す直前にのみ行い、DBには影響させない

**代替案**: DBのテキストを変換済みで保存 → 辞書更新時に全セグメントを再変換する必要があり複雑

### 決定3: 変換アルゴリズム

**採用**: 辞書エントリを表記の長さ降順にソートし、テキストを先頭から走査して最長一致で置換する（`String.replaceAll` の直列適用ではなく、単一パスで最長優先）。

**理由**: 「山田太郎」→「やまだたろう」が登録されている場合、「山田」→「やまだ」で部分一致してしまうことを防ぐ。

**代替案**: `String.replaceAll` の直列適用 → 短い表記が長い表記の部分に先に一致する問題がある

### 決定4: UIの配置

**採用**: TTS編集ダイアログの上部に「辞書」ボタンを追加し、クリックで `TtsDictionaryDialog` を別ダイアログとして表示する。

**理由**: 辞書管理はセグメント編集とは独立した操作であり、常時表示するとUIが複雑になる。必要なときだけ開くダイアログ形式が適切。

## Risks / Trade-offs

- **[リスク] 変換の誤適用**: 一般的な単語の表記が意図せず登録された変換ルールにマッチする → 辞書UIでユーザーが確認・削除できる設計で軽減
- **[リスク] パフォーマンス**: 辞書エントリ数が増えるとテキスト変換のコストが増加 → 通常の小説辞書は数十〜数百エントリ程度であり実用上問題なし

## Migration Plan

- 新規DBファイルのため既存データへの影響なし
- `TtsDictionaryDatabase` は `TtsAudioDatabase` と同様にフォルダパスを受け取ってDBを開く形式
