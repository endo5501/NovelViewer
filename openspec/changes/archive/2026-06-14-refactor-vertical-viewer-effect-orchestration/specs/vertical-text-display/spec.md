## ADDED Requirements

### Requirement: ページ効果の決定論的オーケストレーション

`VerticalTextViewer` がレイアウト確定後に起こすページ単位の副作用——ターゲット行へのページ飛び、ファイル開始位置 intent（`fromEnd`）による最終ページ飛び、TTS ハイライト位置への自動ページ遷移、現在ページ先頭行のレポート、レイアウト変化時のページ遷移アニメーション停止——は、レイアウト結果と現在の状態から導出される決定として扱われ、その適用は決定論的でなければならない（SHALL）。

レイアウト確定までに `build` が複数回実行されても、各ワンショットのページ効果（ターゲットページ飛び、最終ページ飛び、TTS 自動ナビ）は高々 1 回だけ適用されなければならない（SHALL）。ワンショット効果のトリガとなる状態（最終ページ飛び要求、保留中 TTS オフセット）の消費（クリア）は、効果適用と同じ単一の適用箇所で行われなければならず（SHALL）、レイアウト計算の最中に行ってはならない（SHALL NOT）。

同一 `build` で複数のページ飛び要因が同時に成立した場合、それらは決定論的な固定順（ターゲット行ページ飛び ① の後に最終ページ飛び ③）で適用されなければならず（SHALL）、最終的な表示ページは本リファクタ前の実装と同一でなければならない（SHALL）。レイアウト制約が前回と変化した場合のページ遷移アニメーション停止は、その `build` 内で即座に適用されなければならない（SHALL）。

これらの効果の決定（どの効果をどのページに対して起こすか）は、widget ツリーに依存しない純粋な関数として、不変の入力スナップショット（総ページ数、ターゲットページ、現在ページ、保留中 TTS オフセット、最終ページ飛び要求、ページ別文字オフセット、ページ別先頭行、制約変化の有無、アニメーション進行中か等）から導出可能でなければならない（SHALL）。本要件はページ効果の決定論とワンショット性を規定するものであり、既存のページネーション計算・ナビゲーション・TTS ハイライトオフセット写像の観測可能な挙動は変更しない（SHALL NOT）。

#### Scenario: ワンショットページ効果は build 複数回でも 1 回だけ適用される

- **WHEN** `fromEnd` intent または TTS 自動ナビが保留された状態で、レイアウト確定までにビューアが複数回 `build` を実行する
- **THEN** 対応するページ飛びは高々 1 回だけ適用され、消費後の `build` では同じ効果が再度スケジュールされない

#### Scenario: 同一 build で複数のページ飛びが成立した場合は従来順で適用される

- **WHEN** 同一の `build` でターゲット行によるページ飛び（①）と `fromEnd` による最終ページ飛び（③）が同時に成立する
- **THEN** 両者は決定論的な固定順（① の後に ③）で適用され、最終的な表示ページは従来実装と同じく最終ページになる

#### Scenario: レイアウト制約変化時のアニメーション停止は即時適用

- **WHEN** ページ遷移アニメーション進行中にレイアウト制約が前回と変化する
- **THEN** 進行中のアニメーションはその `build` 内で即座に停止され、繰り越しの遷移は残らない

#### Scenario: 効果の決定が純関数として導出可能

- **WHEN** 総ページ数・ターゲットページ・現在ページ・保留中 TTS オフセット・最終ページ飛び要求・ページ別文字オフセット・ページ別先頭行・制約変化の有無を含む不変スナップショットが与えられる
- **THEN** 起こすべきページ効果（即時ジャンプ先 / アニメ付きジャンプ先 / レポートすべき行 / アニメ停止要否 / hover 非表示要否 / 消費すべきワンショットフラグ）が widget ツリーに依存せず一意に決定できる

#### Scenario: 既存のページ挙動は不変

- **WHEN** ユーザがページ送り、ファイル切替（`fromStart`/`fromEnd`）、TTS 追従、ターゲット行ジャンプを行う
- **THEN** 表示されるページ、ページ遷移アニメーション、初期ページ、先頭行レポートは本要件導入前と同一である
