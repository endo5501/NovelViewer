## Context

リリースのバージョンは「git タグ」と「`pubspec.yaml` の `version`」の 2 系統から決まる。

- `release.yml` はタグ名（`${{ github.ref_name }}`）から `v` を除いた値で Release 名・インストーラ・ZIP 名を決める。
- アプリが名乗る `currentVersion` は `PackageInfo.fromPlatform().version`（= ビルド時に `pubspec.yaml` から焼き込まれる値）であり、`update_providers.dart:42` で更新判定に渡される。
- 更新判定 `isNewer`（`version_comparator.dart`）は `_stripBuild` でビルドメタ（`+N`）を除去して比較するため、ビルド番号 `N` は更新判定に影響しない。

両者の一致を保証する仕組みがないため、`pubspec.yaml` の更新を忘れてタグだけ push すると、Release 名は新バージョン・バイナリは旧バージョンという壊れた Release が公開され、インストール済みアプリが永続的に「アップデートあり」を出し続ける。

開発・push は Windows 11 環境で行われる。既存スクリプトは `scripts/*_macos.sh` と `scripts/*_windows.bat` のように OS 別に分かれている。`release.yml` は多数の `throw`/`exit` ガードを持つスタイルで書かれている。

## Goals / Non-Goals

**Goals:**
- `pubspec.yaml` の更新忘れによる不整合リリースを、push 前（主防御）と公開前（保険）の二段で防ぐ。
- リリース手順をワンコマンド化し、手作業の余地（＝忘れる余地）を減らす。
- Windows / Unix 双方のシェルからリリースできるようにする。

**Non-Goals:**
- バージョン体系そのものの変更（タグ単一ソース化など）は行わない。`pubspec.yaml` を真実の源として維持する。
- 更新判定ロジック（`lib/features/app_update/`）には手を入れない。
- macOS リリースの自動化フローはこの変更の対象外（既存のまま）。

## Decisions

### 決定1: 二段防御（ローカルスクリプト＋CI 検証）
- **採用**: ローカル release スクリプトを主防御、`release.yml` の検証ステップを保険にする。
- **理由**: ローカルだけだと「スクリプトを使わず手打ち tag」で素通りする。CI だけだとフィードバックが push 後になり、不正タグを毎回消す手間が出る。両方で「速いフィードバック」と「公開前の最終防御」を両取りする。
- **却下案**: タグを単一ソースにして `flutter build --build-name/--build-number` で注入する案。よりエレガントだがリリース思想の変更とビルド番号運用の決定が必要で、スコープが広がるため今回は見送り。

### 決定2: スクリプトは PowerShell と bash の両方を提供
- **採用**: `scripts/release.ps1` と `scripts/release.sh` を用意し、ロジックを揃える。
- **理由**: 主環境は Windows（`.ps1` が素直）だが、Git Bash / WSL / mac でも使えるよう `.sh` も用意する。ユーザ選択により両方提供で合意済み。
- **トレードオフ**: 同一ロジックを 2 実装するため保守コストは倍。検証順序・メッセージ・終了コードの仕様を spec で固定し、ドリフトを防ぐ。

### 決定3: 事前検証はすべて「変更前」にまとめて行う
- **採用**: 形式・clean・ブランチ・タグ未使用・後退でないことを、`pubspec.yaml` を触る前に全部チェックし、1 つでも失敗したら何もせず非ゼロ終了する。
- **理由**: 途中まで進んでから失敗すると、commit 済み・tag 済みの中途半端な状態の復旧が面倒。「全部 OK なら進む、ダメなら一切触らない」を保証する。

### 決定4: ビルド番号は現値 +1（単調増加）
- **採用**: `pubspec.yaml` の現ビルド番号を読み、+1 した値を書き込む。
- **理由**: `isNewer` がビルドメタを捨てるため更新判定には無関係。Flutter 慣習どおり単調増加させておけば表示・識別用途で十分。バージョンごとにリセットはしない。

### 決定5: CI 検証ステップは専用スクリプトに切り出す
- **採用**: 検証ロジックを `scripts/verify_release_version.sh <tag>` に切り出し、`release.yml` の checkout 直後に `shell: bash` で呼ぶ。スクリプトはタグ名から先頭 `v` を除いた値と、ビルドメタを除いた `pubspec.yaml` の version を比較し、不一致なら `exit 1`。
- **理由**: ビルドを伴うどのステップよりも前に落とすことで計算資源を使わず即座に失敗させる。インラインの YAML シェルは単体テストできないが、スクリプト化すれば TDD（タグ==pubspec の比較・ビルドメタ無視）でカバーできる。windows-latest には Git Bash が同梱されるため `shell: bash` が使える。
- **却下案**: `${GITHUB_REF_NAME#v}` を YAML にインライン記述する案。短いが検証ロジックをテストできず、`.sh`/`.ps1`/CI で同じ比較規則がドリフトしやすい。

## Risks / Trade-offs

- [スクリプトを使わず手打ち tag される] → CI 検証ステップ（保険）が公開前に必ず止める。二段防御の意義そのもの。
- [`.ps1` と `.sh` のロジックがドリフトする] → 検証項目・順序・終了コード・ビルド番号計算を spec のシナリオで固定し、両実装が同じテストケースを満たすことを確認する。
- [`pubspec.yaml` の version 行のパースが脆い] → `version:` 行の `X.Y.Z+N` を素直に正規表現で抽出する。形式が想定外なら検証で落とす方針にし、誤った書き換えを避ける。
- [push まで自動化することの副作用（誤実行で即リリース）] → 事前検証（clean / main / タグ未使用 / 後退でない）を厳格にし、想定外の状態では一切 push しない。
- [CI 検証が Windows runner のシェル差異で誤判定] → 比較ロジックはシンプルな文字列処理に留め、シェル依存の少ない実装にする。

## Migration Plan

1. スクリプトと CI 検証ステップを追加（挙動変更なし。既存リリース成果物には影響しない）。
2. 次回リリースから release スクリプト経由の運用に切り替える。
3. ロールバックは追加ファイル削除と `release.yml` の検証ステップ除去のみで完結（破壊的変更なし）。
