## Context

NovelViewerのTTS機能は `third_party/qwen3-tts.cpp` サブモジュールを通じてggmlを利用している。ggmlはネスト2階層のサブモジュール構成:

```
NovelViewer
  └── third_party/qwen3-tts.cpp  (endo5501/qwen3-tts.cpp)
        └── ggml/                 (ggml-org/ggml, 現在 v0.9.6+42)
```

v0.9.6 → v0.9.11間のAPI互換性は調査済みで、公開ヘッダに破壊的変更はない。v0.9.8 → v0.9.11ではヘッダのSHAが完全一致しており、変更は全てバックエンド内部の最適化・バグ修正に限定される。

## Goals / Non-Goals

**Goals:**
- ggmlをv0.9.11に更新し、Metal/Vulkanバックエンドの最適化を取り込む
- 更新前後のパフォーマンスを定量的に測定・記録する
- macOS (Metal) での動作を確認する

**Non-Goals:**
- qwen3-tts.cppのコード変更（API互換のため不要）
- 新しいggml機能（NVFP4量子化、OpenVINOバックエンド等）の活用
- Windows (Vulkan) でのビルド確認（環境依存、別途対応）
- ggmlの量子化モデル対応やモデル変更

## Decisions

### 1. v0.9.11に一括更新（段階的更新ではなく）

**選択**: v0.9.6 → v0.9.11 に一括更新

**理由**:
- v0.9.6→v0.9.8、v0.9.8→v0.9.11の両方でAPI互換性を確認済み
- v0.9.8→v0.9.11は公開ヘッダのSHAが完全一致（リスクゼロ）
- 段階的更新は中間状態のテストコストが増えるだけでメリットがない

**却下した代替案**:
- v0.9.8に更新後、別changeでv0.9.11に更新 → 中間ビルド・テストのオーバーヘッドが無駄

### 2. qwen3-tts.cppリポジトリ側で先にサブモジュール更新

**選択**: qwen3-tts.cppリポジトリでggmlサブモジュールを更新・コミットし、NovelViewerのサブモジュールポインタを後から追従

**理由**:
- ggmlはqwen3-tts.cppのサブモジュールなので、qwen3-tts.cpp側でコミットが必要
- NovelViewerはqwen3-tts.cppの特定コミットを参照するだけ

### 3. ベンチマーク測定の方法

**選択**: 既存の `scripts/benchmark_tts.sh` を使い、同一条件で更新前後を測定

**理由**:
- 既にper-stage計測（tokenize/encode/generate/decode/total/RTF）が実装済み
- JSON出力で定量比較が容易
- warmup + 複数回実行の中央値で信頼性を確保

**測定条件**:
- モデル: 0.6B F16（標準モデル）
- テキスト: デフォルトの日本語テスト文
- warm-up: 1回、計測: 5回
- `--max-tokens 200`（ベンチマーク用の制限値）

## Risks / Trade-offs

- **[動作変化]** バックエンド内部の最適化により、浮動小数点演算の順序変更等で出力音声が微妙に変わる可能性がある → ベンチマーク時に音声出力の品質を聴覚確認する
- **[ビルド失敗]** ggml v0.9.11のCMake構成変更でビルドが通らない可能性 → CMakeオプションに影響する変更は調査済みでなし。ビルド失敗時はCMakeエラーから対処
- **[パフォーマンス劣化]** 更新後にパフォーマンスが低下する可能性 → ベースライン測定で定量比較し、劣化があれば原因調査。最悪の場合はロールバック可能
- **[Windows未確認]** macOSのみの確認となる → Windows対応は必要に応じて別途実施
