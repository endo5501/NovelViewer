## Context

現在のブックマーク機能はファイル単位（`novel_id` + `file_path`）で一意に管理されている。Bookmarkドメインモデルには行番号情報がなく、ブックマークを開くと常にファイルの先頭が表示される。テキストビューアには検索結果からの行ジャンプ機能（`_scrollToLine`）が既に実装されており、この仕組みを再利用できる。

DBは現在バージョン3で、`bookmarks`テーブルのUNIQUE制約は`(novel_id, file_path)`。

## Goals / Non-Goals

**Goals:**

- ブックマークに行番号を記録し、その行へジャンプできるようにする
- 同一ファイル内に複数のブックマークを作成できるようにする
- ブックマークリストに行番号を表示する
- テキストビューア上でブックマーク行に栞マークを視覚的に表示する
- 水平・縦書き両モードでブックマーク行ジャンプに対応する

**Non-Goals:**

- ブックマーク位置の自動更新（ファイル内容が変更された場合の行番号ズレの自動補正）
- ブックマークへのメモ・コメント機能
- ブックマークのエクスポート/インポート機能

## Decisions

### 1. 行番号の定義と取得方法

**決定**: 水平モードではScrollControllerのオフセットから行番号を算出する。縦書きモードでは現在表示中のページ番号を行番号フィールドに記録する。

**理由**: テキストビューアの既存の`_scrollToLine`メソッドが行番号ベースでスクロール位置を計算しており、この仕組みと整合性がある。縦書きモードではページ単位の表示のため、ページ番号を利用する。

**代替案**: 文字オフセットを記録する方式も検討したが、行番号の方がユーザーにとって直感的で、既存のスクロール機構との互換性も高い。

### 2. DB一意制約の変更

**決定**: UNIQUE制約を`(novel_id, file_path)`から`(novel_id, file_path, line_number)`に変更する。

**理由**: 同一ファイルの異なる行にブックマークを設置できるようにするため。同一ファイル・同一行への重複ブックマークは防止する。

**マイグレーション**: SQLiteではALTER TABLEでUNIQUE制約を変更できないため、テーブルを再作成する。既存データは`line_number = NULL`で移行する。

### 3. 栞マークの表示方法

**決定**: 水平モードでは、ブックマーク行の左端にBookmarkアイコンをOverlayまたはStack内で表示する。縦書きモードでは、該当ページの先頭に栞アイコンを表示する。

**理由**: テキスト本文に栞文字を挿入する方式は、テキスト選択やTTSハイライトに影響を与えるため避ける。オーバーレイ方式ならテキストデータに影響なく視覚的に栞マークを表示できる。

### 4. 現在表示行の取得方法

**決定**: 水平モードではScrollControllerの現在位置とフォントサイズ・行高さから表示中の先頭行を算出する。縦書きモードでは現在のページインデックスを使用する。

**理由**: テキストビューアの既存ロジック（`_scrollToLine`、`_scrollToTtsHighlight`）と同じ計算方式で整合性を保つ。

### 5. ブックマーク行ジャンプの実装

**決定**: 既存の`_scrollToLine`と同じメカニズムを使用し、`SelectedSearchMatch`と同様のインターフェースでブックマーク行番号を渡す。縦書きモードでは`targetLineNumber`パラメータを利用する。

**理由**: 既存のジャンプ機構を再利用することで、コード重複を避け、水平・縦書き両モードでの一貫した動作を保証する。

## Risks / Trade-offs

- **[行番号のズレ]** ファイル内容が更新されると記録された行番号がずれる可能性がある → 非ゴールとして明示。ブックマーク自体は有効であり、大まかな位置の目安として機能する
- **[DBマイグレーションの複雑さ]** テーブル再作成が必要 → 既存データを保持しつつ`line_number = NULL`でマイグレーションし、後方互換性を維持
- **[縦書きモードでのページ番号]** ウィンドウサイズ変更でページ番号が変動する → ページ先頭の文字位置も記録する方式も検討したが、初期実装ではページ番号のみとし、十分な精度を確保する
