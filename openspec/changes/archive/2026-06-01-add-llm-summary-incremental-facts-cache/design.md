## Context

`LlmSummaryService.generateSummary` の現状は、(1) 単語を全ファイルから検索し、(2) 結果を `episode <= coveredUpToEpisode` で絞り込み、(3) マッチを単一の `List<String> contexts` に平坦化し、(4) それを `LlmSummaryPipeline.generate` に渡す。パイプラインは context を ~4000 字チャンクに詰めて Stage-1 ファクト抽出（チャンクごとに 1 LLM 呼び出し）、再帰集約、最終要約を実行する。`word_summaries` に永続化されるのは最終的な `summary` 文字列だけで、中間ファクトは破棄される。

平坦化された context リストは複数ファイルにまたがり、チャンクがファイル境界をまたいで詰められるため、メモ化できる安定した単位が存在しない。結果として、再解析のたびに対象内の全ファイルのファクトを再抽出する。頻出キャラクターでは、これが支配的コスト（数十回の LLM 呼び出し）であり、そのほとんどが既に解析済みの話に対する冗長な処理である。

キャッシュ可能な単位は、解析間でも、ネタバレなし/ありトリガ間でも安定している必要がある。ソースファイル（話）がその単位である。ファイルの内容は著者が改稿したときにだけ変わり、「どのファイルが対象か」だけが `covered=5` と `covered=20` の実行で異なる唯一の要素だからである。

## Goals / Non-Goals

**Goals:**
- Stage-1 ファクト抽出を解析間で再利用し、再解析では新規追加・変更された話の分だけコストを払う。
- キャッシュをネタバレなし/ありで共有する（`covered_up_to_episode` ではなくファイルでキー付け）。
- 本文改稿（content hash）と抽出プロンプト変更（prompt version）を検出し、影響を受けたファイルだけを再抽出する。
- 「結果が悪いのでやり直す」再解析を維持する。同一 `covered_up_to_episode` での再解析はフレッシュ抽出を生まなければならない。
- コールドキャッシュ時は現状と同一の解析出力。データ移行なし。`word_summaries` 無変更。

**Non-Goals:**
- 集約・最終要約ステージのキャッシュ（入力が選択スコープに依存するため毎回フレッシュであるべき）。
- スナップショットのセマンティクス、hover popup、履歴 UI、マーク描画の変更。
- 語をまたいだファクト共有（キャッシュは `(folder, word, file)` 単位）。
- 増分集約（組み立てたファクトは毎回再集約する。メモ化するのは Stage-1 のみ）。

## Decisions

### D1: チャンク粒度ではなく、キャッシュ可能な単位＝ファイル粒度でキャッシュする
Stage-1 抽出を「ファイルをまたいで context を 4000 字チャンクに詰める」方式から「ファイル単位で抽出」に変更する。各ファイル自身の context が抽出入力となり、単一ファイルの context がチャンクサイズを超える場合のみそのファイル内で分割する（チャンクごとの結果をそのファイルのファクトに統合）。異なるファイルの context は決して同じチャンクに詰めない。

- **理由**: ファイルを混ぜたチャンクは、1 ファイルが変わったときに無効化できず、スコープが拡大したときに再利用もできない。ファイル粒度なら安定したキー（`folder, word, file`）が得られ、ユーザのメンタルモデル（話数＝エピソード＝ファイル）とも一致する。
- **検討した代替案**: チャンクの正確なテキストのハッシュをキーにするチャンク粒度キャッシュ。却下。対象ファイル集合が変わるたびにチャンク構成がずれるため、キャッシュヒットがまれで無効化も曖昧になる。
- **トレードオフ（許容）**: ファイルをまたぐチャンクパッキングを失う。話ごとの言及が薄い単語ではコールド時に Stage-1 呼び出しが小さく多数になる。本命ケース（context の厚い頻出キャラ）は影響なし。proposal 参照。

### D2: 妥当性 = content_hash と prompt_version の一致
キャッシュ行は、保存された `content_hash` がファイルの現在の全内容のハッシュと等しく、かつ保存された `prompt_version` が現行の抽出プロンプトバージョン定数と等しいときのみ再利用可能。いずれか不一致ならミス扱いで再抽出し、行を上書きする。

- **content_hash** は**ファイル全内容**に対して取る（語の context 窓ではない）。話のどこかが編集されればその話のファクトを無効化する。語から遠い編集でも再抽出する点でやや過剰無効化だが、単純で正しい。ずれる context 窓をハッシュするのは脆い。
- **prompt_version** は抽出プロンプトのフォーマットが変わるたびにバンプする整数定数。古いプロンプトで生成されたファクトが新しいものと混ざらないことを保証する。

### D3: 1 つのセントネル無効化経路で 3 トリガを扱う
`content_hash` に無効セントネル値（**空文字列**）を書き込むと、次回解析で確実にミスになる。この単一メカニズムが以下を担う。
1. **本文改稿** — D2 のハッシュ比較で自動的に処理（セントネル不要。網羅性のため記載）。
2. **プロンプト改善** — `prompt_version` 定数をバンプ（全行に対するグローバルな強制ミス）。
3. **やり直し再解析** — 再解析トリガが実行前に対象 `(folder, word)` 行へセントネルを書き込み、同一スコープ再解析でも再抽出させる。

- **#3 にセントネルを使う理由**: ファイル粒度キャッシュでは、素朴な同一スコープ再解析は各ファイルのキャッシュ済みファクトを再利用し、（非決定的な）最終要約を振り直すだけになり、悪い抽出を直したいというユーザ意図を損なう。先に語のキャッシュを無効化することで、現状の「再解析＝最初からやり直し」のセマンティクスを取り戻す。

### D4: オーケストレーションはサービス、パイプラインは再利用可能なステージに分割
`LlmSummaryService` がキャッシュ参照・組み立てを担う。パイプラインは Stage-1 のファイル単位抽出と、集約＋最終ステージを個別に呼び出せる形で公開し、サービスが「絞り込み済み結果をファイルごとにグループ化 → ファイルごとにヒット or 抽出 → 全ファクトを組み立て → 集約＋要約」を行えるようにする。新規 `FactCacheRepository` が既存 `LlmSummaryRepository` と並んで `fact_cache` テーブルをラップする。

- **検討した代替案**: パイプライン自身をキャッシュ対応にする。却下。永続化の関心をパイプライン（本来は純粋な変換）から切り離し、無効化ポリシーをサービスに決めさせるため。

### D5: カスケード削除、追加マイグレーション
`fact_cache` 行は、対応する `word_summaries` 行が削除されるとき（語単位・フォルダ単位の削除）に合わせて削除する。テーブルは追加マイグレーションで作成し、キャッシュは空で開始するため、アップグレード後の初回解析は全ミスのフルスキャンとなり、現状と同一の結果を生みつつキャッシュを温める。

## Risks / Trade-offs

- **薄い単語でのコールドキャッシュコスト増**（D1） → 許容。文書化済み。プロファイルで問題化したら、ファイルキーを保ったまま同一ファイルの context をパックする方式を再検討。
- **無関係な編集での過剰無効化**（D2 全ファイルハッシュ） → 許容。1 話の再抽出はフルスキャンに比べ十分安い。
- **ハッシュ未取得/誤りによるキャッシュ汚染** → ハッシュは検索が読むのと同じファイル内容から計算する。ファイルが読めない場合はミス扱い（古い可能性のあるファクトを決して出さない）。
- **集約ロスの蓄積**は設計上回避される。メモ化するのは Stage-1 のみで、集約＋最終要約は選択スコープの全組み立て済みファクトに対して毎回フレッシュに実行されるため、後の話が毎回早い段階のファクトを再文脈化できる。
- **キャッシュ/スナップショットの乖離**（スナップショットのないキャッシュ行、またはその逆） → 設計上独立。カスケード削除（D5）が削除の整合性を保ち、キャッシュ行が無ければ単にミスになる。

## 実装後に判明したランタイム不具合（解析トリガ）

本変更は「同じ公開エントリポイントの背後で出荷」を前提にしていたが、新依存の追加により**解析トリガ側に 2 件の既存バグが露呈**した。いずれも修正済み・リグレッションテスト付き。仕様（specs）には影響しない実装レベルの修正。

- **Provider 解決レース**: `llmSummaryServiceProvider` は同期 Provider で、新規 `factCacheRepositoryProvider`（FutureProvider）が未解決の間 `null` を返す。これを事前解決するものが他に無いため、`run()` が `llmClientProvider.future` しか await していないと初回解析が黙って no-op になる。→ `run()` でサービス読み取り前に `llmSummaryRepositoryProvider.future` と `factCacheRepositoryProvider.future` も await（`analysis_runner.dart` / テスト: `analysis_runner_test.dart`）。
- **ポップアップ再解析のメニュー teardown**: `MenuItemButton.onPressed` はメニュー自動クローズ後のポストフレームで遅延実行され、その時点で popup は破棄・State は unmount 済み。コールバック内で `context`/`ref`/`widget` に触れると defunct で例外となり解析が起動しない。→ 必要値を `build()`（mounted 中）でキャプチャしてクロージャで使用（`hover_popup_widget.dart` / 再現テスト: `hover_popup_host_test.dart`）。

教訓: 同期 Provider が非同期依存の `.value` で null を返すパターンでは、読み取り側で**全依存の future を await**する必要がある。

## Decided (旧 Open Questions)

- **セントネル表現**: `content_hash` の強制ミス値は **空文字列** とする（カラムを NOT NULL に保ち、比較を全域化するため）。`NULL` は採用しない。
- **再解析の無効化範囲**: 再解析トリガは、対象 `covered` のファイルだけでなく **その語のキャッシュ全体**（`(folder, word)` の全行）を無効化する。「この要約は間違っている」というユーザ意図は抽出そのものを問題視しているため。

## Migration Plan

1. 追加マイグレーションで `fact_cache` テーブルを作成（DB バージョンのバンプ）。バックフィルなし。
2. ファイル単位パイプライン分割＋サービスオーケストレーションを同じ公開エントリポイントの背後で出荷。空キャッシュでは挙動同一。
3. ロールバック: テーブルを drop / バージョンバンプを revert。`word_summaries` と他機能は無変更。
