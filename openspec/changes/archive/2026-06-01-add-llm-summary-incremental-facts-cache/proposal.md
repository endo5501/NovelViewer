## Why

LLM による単語・語句のまとめ機能は、ユーザが単語を解析するたびに、過去のスナップショットが大半の話をすでにカバーしていても、毎回 1 話目からファクト抽出パイプラインを丸ごと再実行している。多数の話に登場するキャラクターを後の話で再解析すると、すでに処理済みの context チャンクを何十個も LLM に再送することになり、トークン・時間・（ローカルモデルでは）GPU を浪費する。高コストな Stage-1 ファクト抽出は同じ処理の繰り返しであり、本当に新しいのは新規追加された話だけである。

## What Changes

- 話（ファイル）単位の **ファクトキャッシュ** を導入する。`(folder, word, file)` ごとの Stage-1 ファクト抽出結果を保存・再利用し、再解析時には未キャッシュ（または内容が変わった）話のファクトだけを抽出する。
- まとめパイプラインを再構成し、Stage-1 ファクト抽出を複数ファイルの context をまたいで 4000 字チャンクに詰める方式から、**ファイル粒度**（キャッシュ可能な単位）での抽出に変更する。集約・最終要約ステージは変更せず、解析ごとに毎回フレッシュに実行する（入力が選択された `covered_up_to_episode` に依存するため）。
- **content_hash + prompt_version** による無効化を追加する。キャッシュ済みファイルのファクトは、ファイルの現在の content hash と現行の抽出プロンプトバージョンの両方が一致したときのみ再利用し、それ以外はそのファイルを再抽出する。
- **無効セントネル（空文字列）の content_hash** を統一的な強制再収集メカニズムとして用い、1 つの経路で 3 ケースを扱う。本文改稿（なろう/カクヨムの加筆修正）、抽出プロンプトの改善（prompt version のバンプ）、「結果が悪いのでやり直す」再解析操作（その語のキャッシュを無効化）。
- 既存の削除経路（`deleteAllForWord`, `deleteByFolderName`）にファクトキャッシュのカスケード削除を連動させ、キャッシュ行が孤立しないようにする。
- **データ移行は不要**。新キャッシュは空で開始し、アップグレード後の初回解析は全ミスのフルスキャンとなって現状と同一の結果を生みつつキャッシュを温める。`word_summaries` は無変更で完全な後方互換。

許容するトレードオフ: ファイル粒度の Stage-1 はファイルをまたぐチャンクパッキングを失うため、コールドキャッシュ時、話ごとの言及が薄い単語では（小さな）LLM 呼び出しが増える。本命ケース（各話に十分な context を持つ頻出キャラクター）は影響を受けず、まさにキャッシュが効く対象である。

## Capabilities

### New Capabilities
- `llm-summary-fact-cache`: `(folder, word, file)` ごとの Stage-1 抽出ファクトを永続化する。content_hash + prompt_version による妥当性チェック、セントネルによる強制無効化、語/フォルダ削除時のカスケード削除を備える。

### Modified Capabilities
- `llm-summary`: 単語解析は、毎回すべての対象ファイルから抽出する代わりに、ファクトキャッシュとミス分の新規抽出を組み合わせてファクトを組み立てる。同一 `covered_up_to_episode` での再解析は、その語のキャッシュを無効化してフレッシュ抽出を強制する。
- `llm-summary-pipeline`: Stage-1 ファクト抽出をソースファイル単位（キャッシュ可能な単位）で実行する。単一ファイルの context がチャンクサイズを超える場合のみそのファイル内で分割する。再帰集約と最終要約は変更しない。

## Impact

- **Code**: `lib/features/llm_summary/data/llm_summary_service.dart`（キャッシュ参照・組み立てのオーケストレーション）、`lib/features/llm_summary/data/llm_summary_pipeline.dart`（Stage-1 のファイル単位分割）、新規ファクトキャッシュリポジトリ＋テーブル、`lib/features/llm_summary/data/llm_summary_repository.dart`（カスケード削除フック）、`presentation/analysis_runner.dart` の再解析トリガ。
- **Database**: 新規 `fact_cache` テーブル（追加マイグレーション）。`word_summaries` のスキーマは無変更。
- **Behavior**: コールドキャッシュ時は現状と同一の解析結果。キャッシュが温まるにつれ再解析が高速・低コスト化する。キャッシュは `covered_up_to_episode` ではなくファイルでキー付けされるため、ネタバレなし/ありで共有される。
