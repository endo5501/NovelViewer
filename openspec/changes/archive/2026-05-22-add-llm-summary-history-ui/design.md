## Context

NovelViewer は本文中の単語を選択すると、右ペインの LLM 解析パネル(ネタバレあり/なし2タブ)で要約を取得・キャッシュする。キャッシュは `word_summaries` テーブルに `(folder_name, word, summary_type)` 一意キーで格納される。`source_file` カラムはネタバレなしエントリだけ書き込まれており、spoiler エントリは常に `NULL` で保存されている。

現状、ユーザは「過去にどの単語を調べたか」を本文を眺めても把握できず、同じ単語を再分析してしまったり、せっかく取った要約の存在を忘れる。読書体験としても解析結果が分離してしまっている。

## Goals / Non-Goals

**Goals:**

- 本文中で「過去に調べた単語」が視認できる(下線/傍線)
- 縦書き・横書き両モードで同じ意味体系のマークを提供
- 既存のハイライト(検索=黄背景、TTS=緑背景)と衝突しない視覚チャネルを使う
- 左カラムにキャッシュ管理用タブを追加し、削除と本文ジャンプを可能にする
- マークおよび管理画面は現在開いている作品(folder)スコープに閉じる(ブックマークと同じ挙動)

**Non-Goals:**

- スポイラー耐性のためのマーク制御(ユーザは再読を前提とする)
- 形態素解析による厳密な単語境界判定(日本語の事情で現実的でないため、最長一致 + 最小長 2 文字で近似)
- 全作品横断のキャッシュ管理ビュー
- マーク表示 ON/OFF トグル設定(将来必要なら別 change で)
- 既存 spoiler キャッシュへの遡及的 source_file 補完

## Decisions

### D1. マークの粒度: 「単語型」(全出現箇所マーク)

`word_summaries` のキーが `(folder, word, type)` で位置を持たないため、自然に「単語単位」で表現できる。インスタンス単位(選択スパンのみ)は新テーブルが必要かつ、ユーザの「何を調べたかを思い出す」目的に対しては過剰。

### D2. 視覚表現: 下線/傍線、点線=なし、実線=あり

- 横書き: ネタバレなし=点線下線、ネタバレあり=実線下線
- 縦書き: ネタバレなし=点線傍線、ネタバレあり=実線傍線
- 両方ある場合: 実線優先(より情報が多いネタバレありを優先)

既存のハイライト(検索=黄背景、TTS=緑背景)が *背景色* チャネルを使っているのに対し、マークは *罫線* チャネルを使うので重畳可能。色は theme から取得し、ダークモードでも視認できる前景色相当を使う。

**Alternatives considered:**

- 圏点(縦書きで点を文字脇に打つ)+ 傍線 → 圏点は強調用途で既知のため意味の混同リスクがある
- 細い罫線/太い罫線で区別 → 細さの違いは縦書きで識別困難
- 色だけで区別 → ダークモード対応で色数が増えると煩雑

### D3. マッチング: 最長一致 + 最小長 2 文字

日本語には明示的な単語境界が無いため、純粋な部分一致では「アリス」を調べると「メアリス」内も誤マーク。完全な形態素解析は重く依存も増える。妥協点として:

- キャッシュ済み単語のうち2文字未満は本文マーク対象外
- 重なるマッチがある場合は最長のものを採用(例: 「アリスの剣」と「アリス」が両方キャッシュされていたら、「アリスの剣を…」では 5 文字側を採用)
- 同じ範囲に複数の単語が異なるタイプでヒットすることはありえない(キーが word + type で一意のため、ある word に対しタイプを統合的に評価する)

### D4. 管理画面: 左カラム3つ目のタブ

既に左カラムには `ファイル | ブックマーク` のタブ構造がある(`bookmark-ui` spec)。同じパターンで `解析履歴` タブを追加する方が、UI 学習コストが最小。

- スコープ: 今開いている作品(folder)のみ。`currentDirectoryProvider` の値で絞り込み
- 表示順: `updated_at` 降順
- 各エントリ: 単語、タイプバッジ(なし/あり/両)、要約冒頭プレビュー、更新日時
- クリック動作: そのキャッシュの `source_file` を開き、テキスト内の最初の出現箇所までスクロール
- 右クリック: 「削除」コンテキストメニュー(ブックマークと同じ操作モデル)

### D5. ジャンプ先解決と source_file の spoiler 拡張

クリックジャンプは `source_file` に依存する。既存スキーマでは spoiler エントリの `source_file` が常に `NULL` で保存されており、これだとネタバレありキャッシュからジャンプできない。

**決定**: spoiler エントリ保存時にも、解析時に開いていた `currentFileName` を `source_file` に書き込む。

- カラム自体は既に存在するためマイグレーションは不要
- 既存の `NULL` 行は触らず、新規および再解析時のみ書き込まれる
- 履歴 UI 側で `source_file` が `NULL` のエントリは「ジャンプ非対応」として扱う(クリックしても何もしないか、無効状態表示)
- `findSummary` 等の既存読み出しロジックは `source_file` を返すよう変更されている(`WordSummary.fromMap`)。読み出し側の互換は維持

### D6. 削除動作

履歴タブの削除は対象行を 1 件削除する(`(folder, word, type)` のキー指定)。「両方」表示のエントリは内部的に2行存在するため、それぞれ別々に削除可能としたいが、UI のシンプルさを優先して **1 行表示・タイプ別バッジ表示** とし、削除は「この単語の全タイプを削除」にする。

**Alternatives:** タイプごとに別々の行で表示 → リストが膨張、視認性悪化。

### D7. 既存ハイライトとの描画優先順位

同じ文字に複数の効果が乗ったときの優先度:

1. 検索ハイライト(黄背景)
2. TTS ハイライト(緑背景)
3. 解析履歴マーク(下線/傍線 — 背景は変更しない)

マークは背景色を持たないため、1 や 2 と同時表示しても上書きは起こらない。

### D8. パフォーマンス

- ファイル表示時に当該フォルダの `word_summaries` を一括取得 → 2 文字以上の単語のみフィルタ
- 単語リストをハッシュセットに保持し、本文文字列に対し最長一致でマーク区間を計算
- キャッシュ更新時(`saveSummary` / `deleteSummary`)に Riverpod プロバイダ経由で再計算を発火
- 規模目安: 1ファイル ~30K 文字 × ~200 単語 → 数 ms オーダーで完了見込み
- 重い実装(Aho–Corasick 等)は不要、必要になれば後で差し替え

## Risks / Trade-offs

- **[偽陽性マーク(部分一致)]** → 最小長 2 文字フィルタで大幅軽減するが、3 文字以上でも「メアリス」例のような事故は残る。ユーザは履歴タブからエントリを削除すれば回避可能。
- **[既存 spoiler キャッシュは jump 非対応]** → `source_file=NULL` のエントリには非対応バッジを出す。再解析すれば書き込まれる。
- **[「両方」エントリの削除粒度]** → 1単語まとめて削除のみ。タイプ別削除は将来必要になったら追加。
- **[マーク描画コスト]** → 単語数が極端に多い作品(>1000)の場合は再描画コスト懸念。初期実装はナイーブ実装で進め、ボトルネックが見えたら最適化。
- **[ルビ付き文字]** → マークはベース文字側に付与。ルビの選択(現状の selectedText 取得経路)で来る単語は、ベース文字には現れないためマークが付かないケースがある(例: 「せいいん」を選択して保存 → 本文の "聖印" にはマーク付かない)。これは現状の selectedText の取得仕様に従い、必要なら別 change で扱う。
