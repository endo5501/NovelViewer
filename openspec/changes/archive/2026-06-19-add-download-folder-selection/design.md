## Context

NovelViewer のダウンロード経路は、保存先を `outputPath` という単一のパラメータに集約しており、最終的な保存先は `p.join(outputPath, folderName)`（`folderName = {siteType}_{novelId}`）で決まる。現状、`outputPath` には常に `libraryPathProvider`（ライブラリルート）が渡される。

一方、ライブラリ側はすでに小説フォルダのネストを前提に設計されている:
- `isNovelFolder` は leaf名（`folder_name`）のみで判定し、ネストの深さに依存しない。
- `resolveNovelId` / `resolveNovelFolderPath` はパスを上方向に辿り、最初に登録済み folder_name に一致した祖先を小説フォルダとして解決する。
- ファイルブラウザには `moveDirectory`（フォルダ移動）が存在し、小説を整理用サブフォルダへ移せる。

つまり「読み込み・判定・整理」はネスト対応済みで、ダウンロードの保存先だけがルート固定という非対称が残っている。

メタデータDB（`novels` テーブル）は `folder_name`（leaf名、UNIQUE）のみを保持し、物理的なネスト位置は記録しない（意図的な「leaf名主義」設計）。

## Goals / Non-Goals

**Goals:**
- ダウンロードダイアログで、ライブラリルート配下の既存サブフォルダを保存先として選択できる。
- 既定はライブラリルート（現状の動作と完全互換）。
- サブフォルダに保存した小説を「更新」しても、ルート直下に重複せず元の物理位置へ上書きされる。
- スキーマ変更なしで実現する（マイグレーション不要）。

**Non-Goals:**
- ダイアログ内での新規フォルダ作成（既存のファイルブラウザのフォルダ作成機能を使う前提。今回は既存フォルダの選択のみ）。
- ライブラリ外（任意の絶対パス）への保存。保存先はライブラリルート配下に限定する。
- メタデータDBへの物理位置（相対パス）の永続化。
- 既存小説の一括移動や再配置機能。

## Decisions

### 決定1: 保存先はダイアログでの選択値を `outputPath` として渡す

保存先選択UIをダウンロードダイアログに追加し、選択値（ライブラリルート、またはルート配下のサブフォルダの絶対パス）を `startDownload(outputPath:)` に渡す。`startDownload` 以下のロジック（`p.join(outputPath, folderName)`）は変更不要。

- **代替案**: ファイルブラウザの「現在開いているフォルダ」を暗黙の保存先にする（探索時のB3案）。却下理由: ユーザーは「ダイアログで明示的に選びたい」と表明した。暗黙挙動は誤保存の発見が遅れる。

### 決定2: 選択候補は「整理用フォルダ（非小説フォルダ）」のみ

保存先候補は、ライブラリルート配下のディレクトリのうち **小説フォルダ（登録済み folder_name）を除いた整理用フォルダ** を再帰的に列挙して提示する。小説フォルダ配下への保存は小説の入れ子を生むため候補から除外する。先頭に「ライブラリルート（既定）」を置く。

- 列挙は `allNovelsProvider` の folder_name 集合と `isNovelFolder` を用い、小説フォルダはそのサブツリーごと候補から外す。
- 表示は相対パス（例: `完結済み/異世界`）。
- **代替案**: 直下1階層のみ（`listSubdirectories` をそのまま使用）。却下理由: 既存の深いネストに保存できず、機能が中途半端になる。再帰列挙でネスト対応を一貫させる。

### 決定3: refresh は呼び出し元の物理パスから保存先を解決する

`refreshNovel` の唯一の呼び出し元 `file_browser_panel.dart` は対象 `DirectoryEntry`（`dir.path` = 物理フルパス）を保持している。`refreshNovel` に対象小説フォルダの親ディレクトリを渡せるよう拡張し、`startDownload(outputPath: <親ディレクトリ>)` とする。これにより小説は現在の物理位置へ上書きされる。

- メタデータの `folder_name` は不変（leaf名は変わらない）。`p.join(outputPath, folderName)` が現在位置に一致する。
- **代替案A**: `novels` テーブルに相対パスを追加（v10マイグレーション）。却下理由: 「物理位置をDBに記録しない」既存設計に逆行し、移動操作との同期責務が増える。
- **代替案B**: refresh時にライブラリ木を走査し leaf名一致を探す。却下理由: 呼び出し元が既に物理パスを持つ現状では不要なI/Oと同名衝突リスク。将来パスを持たない呼び出し元が増えた場合の fallback としてのみ `resolveNovelFolderPath` を検討余地に残す。

### 決定4: スキーマ変更なし

物理位置はDBに保存しない。保存先はダウンロード時に明示選択され、refreshは呼び出し元の物理パスで解決されるため、永続化は不要。

## Risks / Trade-offs

- [refreshの呼び出し元がパスを持たないケースが将来増えると重複ダウンロードが再発する] → 当面の呼び出し元は1箇所（パス保持済み）。`refreshNovel` のシグネチャで保存先を必須・明示にし、将来の呼び出し元にもパス指定を強制する。必要なら `resolveNovelFolderPath` による fallback を後続変更で追加。
- [選択フォルダがダイアログ表示中に外部から削除される] → ダウンロード開始時に保存先ディレクトリを `createNovelDirectory` が `recursive` 生成するため、消えていても再生成される。ライブラリ外の親が消えるケースは想定外（ライブラリ配下限定）。
- [候補列挙が大きなライブラリで遅い] → 整理用フォルダのみ（小説フォルダのサブツリーは枝刈り）の再帰列挙であり、通常規模では問題にならない。必要なら遅延読み込み・キャッシュを検討。
- [サブフォルダ保存により、ユーザーがファイルブラウザのルートで小説を見つけられなくなる] → これは整理機能の意図どおり。既存の移動機能と同じ前提。
