## Context

NovelViewerはダウンロードした小説のメタデータ（タイトル含む）をSQLiteのnovelsテーブルに保存している。タイトルはダウンロード時にWebサイトから自動取得されるが、ユーザーが手動で変更する手段がない。ファイルブラウザのコンテキストメニューには既に「更新」「削除」があり、同様のパターンで「タイトル変更」を追加できる。

## Goals / Non-Goals

**Goals:**
- ユーザーがファイルブラウザのコンテキストメニューからタイトルを変更できる
- 変更後のタイトルがDB（novelsテーブル）に永続化される
- タイトル変更後、ファイルブラウザの表示が即座に更新される

**Non-Goals:**
- フォルダ名（narou_<ID>形式）の変更は行わない（DBのtitleフィールドのみ変更）
- 「更新」（再ダウンロード）時のタイトル上書き防止（現状の動作を維持し、更新時はWebサイトのタイトルで上書きされる）
- バッチタイトル変更（一括変更）

## Decisions

### 1. コンテキストメニューへの追加位置
「タイトル変更」は「更新」と「削除」の間に配置する。タイトル変更は破壊的操作ではなく、頻度として更新と削除の間程度と想定。

### 2. タイトル変更UIはダイアログ形式
TextFieldを含むAlertDialogを使用する。現在のタイトルをプリフィルし、ユーザーが編集して「変更」ボタンで確定する。既存の削除確認ダイアログと同様のパターンを踏襲。

**代替案**: インラインエディット（ListTileのタイトル部分を直接編集可能にする） → 実装が複雑で既存パターンから外れるため却下。

### 3. NovelRepositoryにupdateTitleメソッドを追加
folderNameをキーとしてtitleフィールドのみを更新する専用メソッドを追加する。既存のupsertメソッドはダウンロード結果全体を更新するため、タイトル単体の更新には適さない。

### 4. バリデーション
空文字列のタイトルは許可しない。入力が空の場合は「変更」ボタンを無効化する。

## Risks / Trade-offs

- [リスク] 「更新」操作でWebサイトのタイトルに上書きされる → ユーザーが手動変更したタイトルが失われる可能性がある。初期リリースではこの動作を許容し、将来的に「カスタムタイトルフラグ」の追加を検討。
- [リスク] 長すぎるタイトルの入力 → UIレイアウトに影響する可能性があるが、現状のダウンロードタイトルでも長いものがあるため、既存の表示ロジックで対処可能。
