## Context

用語要約は2段階のLLMパイプラインで生成される。Stage1（事実抽出）は本文の文脈チャンクから箇条書きの事実を抽出し、Stage2（最終要約）はその事実を1〜2文に要約する。ユーザが目にするのはStage2の出力のみで、Stage1の事実は中間生成物として `fact_cache`（キー: `folder, word, file`）にキャッシュされ、最終要約は `word_summaries` テーブル（キー: `word, covered_up_to_episode`）に永続化される。

現状、プロンプトを構築する `LlmPromptBuilder` は指示文・出力フォーマットがすべて日本語リテラルでハードコードされており、出力言語を指定する文も存在しない。表示言語設定（`localeProvider`、対応言語 `ja/en/zh`）はUIのl10nにしか使われておらず、プロンプト生成経路には伝播していない。このため、表示言語を英語・中国語にしても要約は日本語で生成される。

本文は必ずしも日本語ではない（ユーザが英語・中国語のテキストファイルを配置しうる）。

## Goals / Non-Goals

**Goals:**
- Stage1・Stage2の両プロンプトの出力言語を、UI表示言語（ja/en/zh）に追従させる。
- 固有名詞（作品中の人名・地名など）は翻訳せず原語のまま維持する。
- `localeProvider` の言語コードをプロンプト生成地点まで配線する。

**Non-Goals:**
- キャッシュ（`fact_cache`／`word_summaries`）への言語次元の追加（言語カラム・キー拡張・マイグレーション）は行わない。
- 言語切替時の既存キャッシュの自動無効化・自動再生成は行わない。
- 本文の言語自動判定は行わない（出力言語はあくまでUI設定に従う）。
- `ja/en/zh` 以外の言語対応は対象外。

## Decisions

### 決定1: Stage1・Stage2の両方をUI表示言語で出力する

事実抽出だけ日本語固定とし最終要約のみ表示言語にする案も検討したが、本文が日本語とは限らないため「Stage1は日本語が高精度」という前提が成立しない。両ステージを一貫してUI表示言語に揃える方が単純で、ステージ間の言語食い違いによる翻訳段差も避けられる。

### 決定2: 固有名詞は原語のまま維持する

固有名詞を表示言語へ訳すと、表記ゆれや作品内同一性の混乱を招く。両プロンプトに「作品固有の人名・地名などの固有名詞は原語のまま記載する」旨を明示する。

### 決定3: 言語コードを引数で配線する（グローバル参照を避ける）

`localeProvider` はRiverpodの `NotifierProvider<LocaleNotifier, Locale>` で、`AnalysisRunner` は既に `_ref.read(...)` で各種プロバイダを参照している。`AnalysisRunner` で `localeProvider` の言語コードを読み取り、`generateSummary(language:)` → `LlmSummaryPipeline(language:)` → `LlmPromptBuilder.build*(language:)` と引数で明示的に渡す。`LlmPromptBuilder` はステートレスな純粋関数のままとし、テスト容易性を保つ。

代替案として `LlmPromptBuilder` 内でグローバルにlocaleを参照する案もあるが、純粋関数性を損ない単体テストが難しくなるため採らない。

### 決定4: キャッシュはスキーマ変更しない／`currentPromptVersion` はbumpしない

言語切替で既存キャッシュと表示言語が食い違っても、要約は「小説 - 単語」単位でしか管理されておらず、不都合があれば既存の単語削除UI（`deleteAllForWord`、履歴パネルの単語削除）で削除・再生成すれば足りる。言語カラム追加やマイグレーション（v10）はコスト過大なため行わない。

`FactCacheRepository.currentPromptVersion` をbumpすると既存fact_cacheが一括無効化され次回解析で自動再抽出されるが、最終要約（`word_summaries`）は `currentPromptVersion` の対象外で手動削除が必要なまま。片側だけ自動更新されるのは一貫性を欠くため、bumpせず「不都合があれば手動削除」で統一する。

## Risks / Trade-offs

- [言語切替後に古い言語のキャッシュがヒットする] → 仕様上の割り切り。既存の単語削除UIで再生成可能であることをもって許容する。proposal/specに運用として明記。
- [Stage1とStage2で生成タイミングが異なり、途中で言語設定を変えると事実と要約の言語が混在しうる] → 解析は通常一連の操作で完了し、設定変更を跨ぐのは稀。混在した場合も削除・再生成で解消できるため許容する。
- [プロンプト文面変更により既存fact_cacheの内容（日本語事実）と新プロンプトの想定がずれる] → `currentPromptVersion` をbumpしないため既存キャッシュは残るが、出力フォーマット（JSON構造）は不変なので破綻しない。言語不一致は決定4の運用で吸収する。
- [LLMが固有名詞維持の指示に従わない可能性] → プロンプトで明示することで最善を尽くす。完全な保証はモデル依存であり本変更の範囲外とする。
